# RFC 20260301202926: Emergency fix for Singleton Crash on Windows

## Introduction
Immediately following the release of v1.1.2, a critical issue was discovered where the Yomitan API server would fail to start with the error "Native host has exited" in the browser logs. Investigation revealed that on some Windows configurations, `os.kill(pid, 0)` can raise a `SystemError` (wrapping `[WinError 87]`) instead of the expected `OSError` or `ProcessLookupError`.

## Objectives
- Prevent the server from crashing on startup when performing the single-instance check.
- Ensure the server can handle platform-specific process management errors gracefully.

## Implementation Details

### 1. Robust Error Handling in `ensure_single_instance`
The error handling block in `ensure_single_instance` was broadened from a specific subset of exceptions to a general `except Exception`.
- **Why**: Windows-specific crashes often manifest as `SystemError` or other internal Python errors when interacting with system PIDs that are in a transitional state or protected.
- **Outcome**: If any error occurs while checking if an old instance is alive or trying to kill it, the server now simply proceeds to claim the instance lock for itself, avoiding a startup crash.

## Analytics and Decisions
- **Safety First**: Since `ensure_single_instance` is an optimization to prevent port conflicts, it should never be a fatal point of failure that prevents the server from starting at all. Broadening the catch block ensures high availability.
- **Traceability**: The crash was identified via `error.log` which captured the `SystemError` traceback.

## Verification
- **Unit Test**: Added `test_no_crash_on_system_error` to `tests/test_server.py`. This test mocks `os.kill` to raise a `SystemError` and verifies that the function catches it and successfully writes the new PID to the crowbar file.
