# RFC 20260301221919: Singleton Robustness Fixes

- **Status**: Staging
- **Request ZIDs**: 20260301221919, 20260301222531
- **Release**: v1.1.4

## Problem Statement

Two edge cases in the `ensure_single_instance` logic undermined the singleton protection of the Yomitan API server:

1. **Lock File Leakage**: When a second instance of the API was launched (e.g., manually in a TTY), it correctly detected the existing process and exited. However, its `finally` block would unconditionally call `delete_crowbarfile()`, deleting the lock file belonging to the *active* background instance. This left the system without a lock, allowing subsequent launches to start conflicting background processes.
2. **Empty Lock Bypass**: If the `.crowbar` file existed but was empty (e.g., due to a disk full error or crash during write), `ensure_single_instance()` would return early without writing its own PID. This resulted in a running server without a valid lock marker, again allowing race conditions.

## Implementation Details

### 1. PID-Aware Deletion
The `delete_crowbarfile()` function was modified to be "PID-aware". It now reads the content of the `.crowbar` file and only performs the `os.remove()` if the PID inside matches the current process ID.

**Analytics & Decisions:**
- Decided against using `fcntl` or `msvcrt` locking to keep the code simple and portable across Windows/Unix without external dependencies.
- Added a specific "close before delete" step to avoid `PermissionError` on Windows systems where a file cannot be deleted while a read handle is open (even briefly).

### 2. Overwriting Empty/Invalid Locks
The early return for `not content` in `ensure_single_instance()` was removed. If the file is empty or contains non-integer data (handled by the `Exception` block), the code now falls through to the standard PID-writing logic.

### 3. Regression Testing
Expanded `tests/test_server.py` with:
- `test_deletes_existing_file_if_owned_by_current_process`: Verified happy path.
- `test_does_not_delete_if_owned_by_other_process`: Verified the fix for lock leakage.
- `test_overwrites_empty_crowbar`: Verified that empty files no longer cause bypasses.
- Updated `test_ensure_single_instance_tty_exits` to simulate the full exit lifecycle (including `finally`).

## Conclusion
These changes ensure that the `.crowbar` lock file accurately reflects the state of the system and that only the process that successfully acquired the lock is allowed to release it.
